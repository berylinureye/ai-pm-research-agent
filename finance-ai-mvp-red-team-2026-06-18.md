# AI 股票研究摘要助手：MVP PRD 与反向质疑

日期：2026-06-18

使用技能：`create-prd` + `strategy-red-team`

体验主题：AI 股票研究摘要助手

边界：本记录只讨论公开信息整理、风险提示和学习型研究摘要，不做荐股、不预测收益、不提供买入或卖出建议。

## 1. 本次目标

前面已经完成三类体验：

- `market-segments`：确定第一轮用户不是普通投资者，而是我自己这个“AI PM 转行者 / 金融背景学习者”。
- `competitor-analysis`：区分 FinRobot、TradingAgents、ai-hedge-fund、通用 LLM 等不同形态。
- `value-proposition`：把价值主张从“帮你赚钱”拉回到“理解公开信息、降低研究成本、标注风险、沉淀作品集”。

这次目标是把它收束成一个非常轻的 MVP：

> 一个面向我自己的金融 AI 项目阅读与研究摘要模板，用来分析热门金融 AI GitHub 项目，而不是做股票投资工具。

## 2. PRD

### 2.1 摘要

AI 股票研究摘要助手的第一版不是一个完整软件产品，而是一个可复用的研究模板和半自动工作流。它帮助我分析金融 AI 项目或股票研究类工具，输出项目类型、目标用户、风险边界、事实 / 推断 / 待验证、作品集启发。

第一版只服务我的转行学习和作品集沉淀，不面向真实投资用户，不输出任何买卖建议。

### 2.2 相关角色

| 姓名 / 角色 | 职责 | 备注 |
| --- | --- | --- |
| 陈卓欣 | 第一用户 / 产品负责人 | 使用该模板训练 AI PM 判断，并沉淀 GitHub 作品集 |
| Codex | 研究和文档助手 | 帮助整理公开资料、运行 skills、生成初稿 |
| 外部读者 | 面试官 / 同学 / 朋友 | 通过 GitHub 看到我的产品思考过程 |

### 2.3 背景

金融 AI 项目近期很热，尤其是 FinRobot、TradingAgents、ai-hedge-fund 这类多 Agent / 金融分析 / 投资模拟项目。但它们容易让没有深度投研经验的人误以为“AI 能直接帮我投资”。

作为转 AI PM 的学习者，我更需要训练的是：

- 如何进入一个高风险行业场景。
- 如何区分项目类型。
- 如何识别产品边界。
- 如何判断公开资料能确认什么、哪些只是推断。
- 如何把 GitHub 项目分析转成作品集材料。

这比做一个“荐股工具”更适合当前阶段。

### 2.4 目标

目标：

> 在 1 周内，用模板分析 2 到 3 个金融 AI 项目，并产出可放入 GitHub 的产品分析记录。

关键结果：

- 完成 2 到 3 个项目的结构化分析。
- 每个项目都明确标注：项目类型、目标用户、核心场景、风险边界。
- 每个项目都区分：已验证事实、合理推断、待验证问题。
- 至少产出 1 个可复用模板。
- 能清楚说出“研究摘要助手”和“荐股助手”的区别。

### 2.5 目标用户

第一用户：

> 我自己：有金融 / 数据背景、正在转 AI PM，但没有丰富个人股票投资经验的人。

第一版不服务：

- 希望 AI 推荐股票的个人投资者。
- 专业投研分析师。
- 自动交易用户。
- 需要真实资金决策的人。

约束：

- 用户不一定懂深度投研。
- 用户需要学习金融 AI 产品边界。
- 用户需要产出作品集材料，而不是投资建议。

### 2.6 价值主张

对于正在转 AI PM、希望用金融 AI 场景训练产品判断的人，这个助手可以把热门金融 AI 项目整理成结构化研究摘要，帮助用户识别项目类型、风险边界和作品集机会，而不是提供投资建议或收益预测。

### 2.7 解决方案

#### 2.7.1 用户流程

1. 选择一个金融 AI 项目或产品，例如 FinRobot。
2. 收集公开资料：README、官网、文档、demo、免责声明。
3. 使用固定模板生成初稿。
4. 人工检查事实和推断。
5. 标注风险和边界。
6. 输出 GitHub Markdown 记录。
7. 列出下一步真实体验任务，例如安装、跑 demo、检查输出来源。

#### 2.7.2 MVP 输出模板

每次分析一个项目，输出以下模块：

1. 一句话说明。
2. 项目类型：研究摘要 / 投资分析 / 交易模拟 / 教育 demo / 企业工作流。
3. 目标用户。
4. 核心使用场景。
5. 解决的痛点。
6. 产品亮点。
7. 技术亮点。
8. 风险和边界。
9. 已验证事实 / 合理推断 / 待验证。
10. 对 AI PM 的启发。
11. 作品集转化方式。
12. 下一步真实体验任务。

#### 2.7.3 必须排除的功能

- 股票推荐。
- 买入 / 卖出建议。
- 收益预测。
- 自动交易。
- 模拟组合收益展示。
- 任何可能让用户误以为“可以直接照做投资”的表达。

#### 2.7.4 技术形态

第一版不开发网页。

第一版只使用：

- Markdown 模板。
- Codex / ChatGPT / Claude 辅助生成。
- GitHub 保存记录。
- 人工复核事实和风险。

如果未来需要轻量工具化，可以再考虑：

- 一个 CLI 模板生成器。
- 一个 Notion / Markdown 模板。
- 一个 Codex skill。

#### 2.7.5 假设

- 我能用公开资料分析金融 AI 项目。
- 结构化模板能提升我的研究效率。
- 事实 / 推断 / 待验证分层能减少误判。
- 这种记录能成为作品集材料。
- 我愿意持续使用这个模板分析 2 到 3 个项目。

### 2.8 发布节奏

版本 0：手动模板

- 用 Markdown 写一份模板。
- 选择 1 个项目试跑。
- 人工检查结果。

版本 1：三项目对比

- 分析 FinRobot、TradingAgents、ai-hedge-fund。
- 做一张对比表。
- 总结不同项目类型和产品边界。

版本 2：作品集化

- 把模板和 3 个案例整理成一个 GitHub 项目 README。
- 补充截图、链接和真实体验任务。
- 写一页“我从金融 AI 项目中学到的 AI PM 产品边界”。

## 3. Red-Team：攻击这个 MVP

### 3.1 最高优先级致命假设

#### 1. 假设：这个模板真的能提升我的学习效率

- **Claim**：结构化模板能让我更快理解金融 AI 项目。
- **Fails if**：我只是把 AI 输出搬进模板，但并没有更理解项目。
- **Evidence to get this week**：选择 FinRobot，用模板分析一次；再手动读 README 30 分钟，比较哪种方式让我提出更多有效问题。
- **Kill criterion**：如果模板没有让我新增至少 5 个明确问题或判断点，就说明它只是漂亮笔记。
- **Cheapest test**：用同一个项目做“模板分析 vs 手动阅读”对比。

#### 2. 假设：公开资料足够支持第一轮产品判断

- **Claim**：README、官网、文档、公开 demo 足够完成第一轮分析。
- **Fails if**：关键判断都必须安装运行后才知道，公开资料只能写空泛介绍。
- **Evidence to get this week**：对 FinRobot、TradingAgents、ai-hedge-fund 各列出 5 条事实、5 条推断、5 条待验证。
- **Kill criterion**：如果每个项目的“待验证”远多于事实，且无法形成任何产品判断，就必须把真实体验提前。
- **Cheapest test**：先不安装，只做三栏事实分层，看可用信息够不够。

#### 3. 假设：这个方向适合我的作品集

- **Claim**：金融 AI 项目分析能体现我的金融 / 数据背景和 AI PM 思考。
- **Fails if**：输出看起来像普通读书笔记，不能体现产品判断、风险边界和落地意识。
- **Evidence to get this week**：把一份分析给 1 到 2 个朋友或 AI 工具评审，问它是否能看出我的产品思考。
- **Kill criterion**：如果外部读者只能看到“我总结了项目”，看不到“我在做产品判断”，就要调整模板。
- **Cheapest test**：在每份分析末尾强制写“对 AI PM 的启发”和“我会怎样设计边界”。

#### 4. 假设：不做荐股的边界能被读者理解

- **Claim**：只要文档写清楚，就不会被误解为投资建议。
- **Fails if**：读者仍然问“那这个股票能买吗”或“哪个项目能赚钱”。
- **Evidence to get this week**：让 2 个不了解项目的人读摘要，问他们认为这个工具是干什么的。
- **Kill criterion**：如果读者把它理解为投资建议工具，就必须重写定位、标题和模板。
- **Cheapest test**：用两个标题做对比：“AI 股票研究助手” vs “金融 AI 项目阅读助手”，看哪个更少误导。

#### 5. 假设：我能持续用这个模板分析多个项目

- **Claim**：我愿意用同一模板分析至少 3 个项目。
- **Fails if**：分析第一篇后就觉得重复、累、没有新收获。
- **Evidence to get this week**：连续分析 2 个项目，记录耗时和新增理解。
- **Kill criterion**：如果第二篇耗时很长且没有新洞察，就说明模板太重。
- **Cheapest test**：给模板设 90 分钟时间盒，超时就删减模块。

### 3.2 目前比较站得住的地方

- 不做荐股、不预测收益，这个边界是正确的。
- 第一用户设为自己，比假装服务大众投资者更稳。
- 先用 Markdown 和 GitHub，不开发网页，符合当前阶段。
- 把事实 / 推断 / 待验证作为默认结构，是相对强的差异化。
- 这个方向能利用我的金融 / 数据背景，但不要求我证明投资能力。

### 3.3 当前无法判断的地方

- 面试官是否会觉得这个主题有价值。
- 我能否真正跑通 FinRobot 或 TradingAgents 的 demo。
- 公开资料是否足够支撑有深度的分析。
- 这个方向是否比其他 AI 应用方向更适合我的作品集。
- 我是否会持续有兴趣分析金融 AI 项目。

## 4. 调整后的最小行动

不要立刻做完整产品。

下一步只做一个 90 分钟测试：

1. 选择 FinRobot。
2. 按模板写一份项目分析。
3. 强制区分事实 / 推断 / 待验证。
4. 写出 5 个真实体验任务。
5. 记录耗时和不懂的地方。

如果 90 分钟内能产出一份比普通总结更有产品判断的文档，就继续分析 TradingAgents。  
如果不能，就说明模板太重或方向还不够清楚，需要缩小范围。

## 5. 本次技能体验评价

### `create-prd` 的帮助

- 它能把前面的用户、价值、竞品分析收束成一个明确方案。
- 它迫使我写清楚目标、用户、非目标和发布节奏。
- 但默认 PRD 模板容易把事情写成“大产品”，需要人工压缩成个人 MVP。

### `strategy-red-team` 的帮助

- 它很适合把“听起来不错”的方案打回现实。
- 它能逼我写清楚失败条件、证据和终止标准。
- 对我这种容易被新方向吸引的人，它能防止继续纸上谈兵。

### 对 AI PM Research Agent 的启发

未来自己的工作流里，PRD 模块必须默认附带 red-team。

否则很容易出现：

- PRD 写得很完整，但根本没人需要。
- 功能看起来合理，但验证成本太高。
- 项目听起来像作品集，但无法体现产品判断。
- AI 输出越来越漂亮，但离真实体验越来越远。
