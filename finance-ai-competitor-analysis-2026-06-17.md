## 3. FinRobot

链接：[FinRobot](https://github.com/ai4finance-foundation/finrobot)

### 一句话说明

FinRobot 是一个面向金融分析的开源 AI Agent 平台，目标是用大模型和多 Agent 工作流自动化金融研究、财务分析和报告生成。

### 目标用户

- 金融研究学习者。
- 量化 / AI 工程开发者。
- 想探索金融 Agent 工作流的研究者。
- 有金融数据分析需求的团队。

### 核心场景

- 读取公司公开信息和财务数据。
- 生成股票研究报告。
- 做公司基本面、新闻、财报相关摘要。
- 通过多 Agent 协作完成不同研究任务。

### 产品亮点

- 明确聚焦金融分析，而不是泛用聊天。
- 能帮助用户从“问一个问题”升级到“生成一份结构化研究报告”。
- 适合学习金融 Agent 如何拆分任务，例如数据获取、分析、报告生成。

### 技术亮点

- Agent 化金融分析工作流。
- 可连接金融数据和 LLM。
- 适合研究“多 Agent + 金融数据 + 报告生成”的架构。

### 风险和边界

- 金融数据准确性和更新时间非常关键。
- 自动生成报告容易给人“专业可信”的错觉。
- 如果没有清楚标注来源、假设和不确定性，容易误导用户。

### 对我的启发

FinRobot 最值得借鉴的是“报告结构化”和“金融任务拆分”。  
但我的第一轮不应照着做专业投研报告，而应做更轻的“公开信息研究摘要”，并强制区分事实、推断和待验证。

## 4. TradingAgents

链接：[TradingAgents](https://github.com/tauricresearch/tradingagents)

### 一句话说明

TradingAgents 是一个多 Agent LLM 金融交易框架，模拟真实交易公司中不同角色的协作，例如基本面分析师、情绪分析师、技术分析师、交易员和风险管理团队。

### 目标用户

- AI Agent 研究者。
- 金融工程 / 量化爱好者。
- 想理解多角色 Agent 协作的开发者。
- 对金融交易工作流感兴趣的学习者。

### 核心场景

- 模拟不同金融角色如何协同分析。
- 研究多 Agent 在复杂决策中的分工。
- 用交易场景测试 LLM agent 的推理和协作能力。

### 产品亮点

- 角色拆分清晰，适合学习复杂工作流如何被 Agent 化。
- 比单一聊天机器人更接近真实金融团队的协作方式。
- 有助于理解“为什么一个金融 AI 产品不能只靠一个模型回答”。

### 技术亮点

- 多 Agent 角色编排。
- 不同分析视角之间的协作。
- 风险管理角色进入流程，而不是只关注收益或交易信号。

### 风险和边界

- 交易场景天然高风险。
- 模拟交易公司不等于真实交易能力。
- 如果用户只看到“多 Agent 下结论”，很容易忽略数据质量、回测偏差、交易成本、合规和极端行情。

### 对我的启发

TradingAgents 适合学习“角色化工作流设计”。  
对于 AI PM Research Agent，可以借鉴它的分工思想：

- 信息收集 Agent。
- 事实核验 Agent。
- 风险审查 Agent。
- 产品总结 Agent。

但不能照搬交易决策场景。第一轮应该把“交易”弱化，把“研究摘要 + 风险提示”强化。

## 5. ai-hedge-fund

链接：[ai-hedge-fund](https://github.com/virattt/ai-hedge-fund)

### 一句话说明

ai-hedge-fund 是一个 AI hedge fund 概念验证项目，用多个 AI agent 模拟投资分析和组合决策流程，并明确强调仅用于教育和研究。

### 目标用户

- AI / 金融工程学习者。
- 想理解投资流程自动化的开发者。
- 对多 Agent 投资 demo 感兴趣的人。

### 核心场景

- 学习 AI 如何模拟投资研究流程。
- 观察多个 agent 如何生成分析和决策。
- 作为教育项目理解金融 AI 的可能性和局限。

### 产品亮点

- 名字和概念非常吸引人，容易激发兴趣。
- 适合 demo 和学习。
- 免责声明边界较清楚：教育用途，不用于真实交易。

### 技术亮点

- 多 Agent 组合。
- 投资研究流程模拟。
- 可作为金融 AI 概念验证样本。

### 风险和边界

- “hedge fund”这个名字很容易让普通用户产生过高期待。
- 教育 demo 和真实金融产品之间差距很大。
- 如果拿它做作品集，需要强调“我分析的是产品边界和风险，不是投资表现”。

### 对我的启发

ai-hedge-fund 最值得学习的是：高风险场景必须主动声明教育用途和非投资建议。  
它提醒我：AI 股票研究助手的 README、PRD 和界面文案都必须把边界放在最前面。

## 6. ChatGPT / Claude 强提示词方案

### 一句话说明

通用大模型可以通过强提示词临时扮演股票研究助手，帮用户整理公司信息、新闻、财报摘要和风险点。

### 目标用户

- 普通 AI 用户。
- AI PM 转行者。
- 产品研究者。
- 需要快速生成初稿的人。

### 核心场景

- 让 ChatGPT / Claude 分析一家公司。
- 生成研究摘要、风险列表、竞品对比。
- 把公开信息转成学习笔记。

### 产品亮点

- 上手成本最低。
- 灵活。
- 不需要安装开源项目。
- 适合做第一轮手动体验。

### 技术亮点

- 强上下文理解。
- 可以快速调整输出格式。
- 适合多轮追问。

### 风险和边界

- 没有固定工作流，输出质量高度依赖提示词。
- 如果不联网或没有来源，很容易编造事实。
- 不会默认区分事实、推断和待验证。
- 不会默认做投资风险边界提示。

### 对我的启发

通用 LLM 是最强替代方案。  
如果 AI PM Research Agent 不能比一个好提示词多做“来源标注、事实分层、风险边界和复用模板”，那就没有必要做成独立工作流。

## 7. Anthropic 金融 Agents 信号

参考：[Anthropic financial services agents announcement](https://www.anthropic.com/news/financial-services-agents)

### 一句话说明

Anthropic 面向金融服务推出了一组由 Claude 驱动的 agents，用于市场研究、估值复核、建模、pitch builder、KYC 等金融任务。

### 目标用户

- 企业金融机构。
- 投行、资管、保险、财富管理等金融团队。
- 需要自动化专业金融工作流的企业用户。

### 核心场景

- 市场研究。
- 估值和建模复核。
- 资料整理。
- 投资备忘录或 pitch 材料生成。
- KYC 和合规相关工作流。

### 产品亮点

- 它说明金融 AI 已经不只是“聊天问答”，而是在进入具体金融工作流。
- 它把金融任务拆成多个专业 agent，而不是一个万能助手。
- 它强化了“金融 AI 必须专业化、流程化、可审查”的趋势。

### 对我的启发

这不是我第一轮能复刻的产品，但它证明方向值得观察。

对个人作品集来说，更现实的是做一个缩小版：

> 面向 AI PM 学习者的金融 AI 项目研究摘要模板。

## 8. 竞品对比结论

### 它们共同说明了什么

这些项目和产品信号说明：

- 金融 AI 正在从“聊天问答”走向“任务工作流”。
- 多 Agent 是金融 AI 的常见表达方式。
- 风险管理、审查和引用比普通消费 AI 更重要。
- 高风险领域不能只追求自动化，还要设计边界。

### 它们对我的第一轮体验有什么帮助

| 我需要学习的能力 | 哪个项目最有帮助 | 原因 |
| --- | --- | --- |
| 金融 Agent 工作流 | FinRobot | 更接近金融研究报告生成 |
| 多角色协作 | TradingAgents | 角色拆分清晰 |
| 风险和免责声明 | ai-hedge-fund | 教育用途边界明确 |
| 低成本手动体验 | ChatGPT / Claude | 最容易自己上手 |
| 企业级金融 AI 趋势 | Anthropic 金融 Agents | 展示金融工作流 agent 化方向 |

## 9. 对 AI PM Research Agent 的复用点

未来如果把这些经验复用到 AI PM Research Agent，至少要加入：

1. **项目类型判断**  
   这个项目是研究摘要、投资分析、交易决策、教育 demo，还是企业工作流？

2. **边界提示**  
   如果涉及金融、医疗、法律等高风险场景，必须提示“辅助信息，不替代专业决策”。

3. **事实 / 推断 / 待验证分层**  
   金融场景尤其不能把推断写成事实。

4. **来源和更新时间**  
   金融数据过期会严重影响判断。

5. **作品集角度**  
   对转行者来说，不需要证明自己会投资，而是证明自己能识别高风险 AI 产品的边界和落地条件。

## 11. 本次技能体验评价

### 有用之处

- `competitor-analysis` 很适合帮我把“热门项目”拆成产品类型。
- 它能防止我只看项目名字和 star，而忽略产品边界。
- 它让我看清不同项目的差异：报告生成、交易模拟、教育 demo、企业工作流，不是一回事。

### 不足之处

- 竞品分析仍然需要更细的 README、文档、demo 和代码验证。
- 对金融专业效果不能只靠产品分析判断。
- 如果要写成作品集，需要补充截图、运行结果或真实输出样例。

### 对我自己的使用建议

我应该先通过我和 Codex 跑一版示范，再自己亲手体验一次。

区别是：

- 我跑：更快得到结构化样例，知道最终产物应该长什么样。
- 自己跑：更能体验工具的输入成本、追问成本、哪里不满意、哪里需要改。

所以最佳方式不是二选一，而是：

> Codex 先示范一版，我再自己复跑一版，然后比较差异。
