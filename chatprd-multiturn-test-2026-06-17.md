# ChatPRD 多轮追问测试记录

日期：2026-06-17

工具：[ChatPRD](https://www.chatprd.ai/)

测试目标：验证 ChatPRD 在首轮输出之后，是否能通过追问补足证据意识、竞品判断、验证计划和作品集落地能力。

## 为什么要补这次测试

前两份记录主要是“首轮输出测试”。它们能看出 ChatPRD 是否能快速生成结构化文档，但不能完整代表 ChatPRD 的产品能力。

这次多轮追问主要补测以下能力：

1. 是否能区分事实、推断和待验证内容。
2. 是否能把 PRD 从“完整工具构想”收敛成最小验证计划。
3. 是否能做反向质疑，找出最危险假设。
4. 是否能把 GitHub 技术项目转成产品分析。
5. 是否能把分析继续转化为作品集计划。
6. 是否能承认前一轮分析太泛、缺少证据。

## 测试结构

本次包含三组任务：

| 组别 | 原始任务 | 追问重点 |
| --- | --- | --- |
| 问题 1 | 为 AI PM Research Agent 写 PRD | 假设拆解、2 周 MVP 验证、严厉 red team |
| 问题 2 | 分析 ChatPRD 的目标用户和核心价值 | 事实 / 推断区分、竞品对比、自我批判 |
| 问题 3 | 把 `addyosmani/agent-skills` 转成产品分析报告 | GitHub 项目事实核验、竞品对比、作品集计划 |

## 问题 1：PRD 生成后的追问

### 问题 1-1：区分已明确、待验证假设、需要补充研究

ChatPRD 将前一版 PRD 分成三类：

已明确：

- 产品定位：面向 AI 产品经理、转行者、分析师的分析工具。
- 主要功能设想：生成结构化产品分析、用户细分、价值主张、MVP 方案等。
- 基本业务目标：报告生成、导出、编辑等核心功能。
- 典型使用流程：输入 URL 或关键词，输出结构化报告，可编辑和导出。
- 非 MVP 范围：不做协作、第三方工具深度集成等。

待验证假设：

- AI PM 等用户是否真的缺乏高效一站式分析工具。
- 用户是否有强烈的作品集 / 结构化分析需求。
- 自动分析是否能显著提效。
- 输出模块和表达格式是否符合目标用户心智。
- 用户是否愿意把自动生成内容用于求职或作品集。
- 用户对自动生成内容的质量和信任门槛。

需要补充研究：

- 目标用户当前使用 Notion、手动分析、AI 问答等方案时的真实痛点。
- 自动化分析结果的可接受度和实际体验。
- 数据隐私、分析深度不足等使用阻碍。
- 作品集输出的行业标准和个性化需求。
- 不同用户群体诉求的权重差异。
- 用户愿意为哪些高级模块付费。

我的评价：

- 这一轮明显比首轮 PRD 更有价值，因为它把“看似完整的 PRD”拆成了假设。
- 它仍然没有真正给出验证优先级，比如哪 3 个假设最危险、哪个必须本周先测。
- 对“已明确”的定义有点宽，有些内容其实只是方案设想，不是用户事实。

### 问题 1-2：2 周 MVP 验证计划

ChatPRD 给出了一个偏用户研究和原型验证的 2 周计划：

- 第 1 天：制定用户画像和访谈提纲。
- 第 2 天：梳理现有解决方案和需求假设。
- 第 3-4 天：访谈 5 到 8 个目标用户。
- 第 5 天：整理需求并修订功能列表。
- 第 6 天：设计低保真原型或报告模板。
- 第 7 天：编写核心场景脚本并确认 MVP。
- 第 8-9 天：做用户可用性测试。
- 第 10-11 天：汇总评价并迭代。
- 第 12 天：做二次快测或 A/B 说明。
- 第 13 天：做冷启动推广测试。
- 第 14 天：总结 MVP 验证结论。

它给出的验证标准包括：

- 目标用户是否认可痛点。
- 是否能理解原型。
- 是否愿意继续使用。
- 是否愿意留下联系方式或继续反馈。
- 是否能形成 2 到 3 个真实作品集案例。

我的评价：

- 这是本轮非常有价值的输出。它从“我要开发工具”拉回到“先验证需求”。
- 计划仍然偏理想化，假设你能在 2 周内找到 5 到 8 个合适目标用户。
- 缺少一个更轻量的个人版验证方式，例如先用手动服务代替产品，帮 3 个人做分析并收集反馈。

### 问题 1-3：严厉产品负责人 red team

ChatPRD 找出的 5 个危险假设：

1. 目标用户真的有强烈痛点，并愿意改变现有工作流。
2. 结构化分析输出与作品集格式能被 AI PM 面试场景认可。
3. 用户愿意信任 AI 自动生成的行业分析内容。
4. 输入 URL / 关键词后，自动分析深度足够。
5. 用户愿意为初级 MVP 付出时间，即使只是免费试用。

它给出的低成本验证方法：

- 问卷 + 3 到 5 个电话访谈。
- 把自动生成样例发给 AI PM 或招聘方打分。
- 对比 ChatGPT、Notion 模板和自动化 demo 的输出信任度。
- 让重度 AI PM 指出自动分析最容易遗漏的内容。
- 做一个简单在线样例交互，看用户是否愿意完整体验并留下反馈。

我的评价：

- 这一轮是目前问题 1 中最强的输出。
- 它真正开始像产品负责人，而不是文档助手。
- 但它仍然没有把 5 个风险排序，也没有明确“先测哪个、为什么”。

## 问题 2：ChatPRD 自我分析后的追问

### 问题 2-1：事实、推断、待验证

ChatPRD 将自己前一轮对 ChatPRD 的分析拆成：

已验证事实：

- 市场上存在 PM、创业团队、小型研发团队写 PRD 和文档的需求。
- 通用大模型存在输出风格不一致、非结构化的问题。
- 用户在协作、知识沉淀、结构化表达方面存在痛点。
- 行业内存在产品文档标准化和模板化需求。
- 通用 AI 工具无法原生支持团队级内容流程管理和历史文档复用。

合理推断：

- ChatPRD 的目标用户包括 PM、创业者、小型团队、技术负责人、敏捷咨询师。
- 这些用户对结构化、自动化、标准化输出 PRD 有需求。
- ChatPRD 相较通用 AI 工具有结构化产出和协同追溯差异化。
- 作品集 / 面试场景对结构化内容有需求。

需要进一步验证：

- 核心用户是否愿意改变现有工作流并持续使用 ChatPRD。
- 结构化输出是否被 AI PM 或招聘方认可。
- 用户是否信任 AI 生成的专业内容。
- 输入 URL / 关键词下的自动分析深度是否足够。
- ChatPRD 是否能留住专业用户，而不仅是尝鲜用户。

我的评价：

- 它有意识地承认大量判断只是推断，这是好现象。
- 但“已验证事实”部分仍然过宽，很多其实是行业常识，而不是针对 ChatPRD 的产品事实。
- 它没有引用官网、定价页、实际功能截图或真实用户反馈，证据层级还不够。

### 问题 2-2：竞品对比

ChatPRD 对比了：

- ChatPRD
- ChatGPT
- Claude
- Notion AI
- pm-skills

对比维度包括：

- 目标用户
- 核心使用场景
- 输出质量
- 工作流能力
- 免费版限制
- 差异化亮点

它的主要结论：

- ChatPRD 聚焦 PRD 和需求分析，行业定制化程度高。
- ChatGPT / Claude 灵活但缺少产品领域标准流程。
- Notion AI 在团队知识库和协作上强，但 PM 深度不足。
- pm-skills 适合产品练习和作品集训练，但自动化和协作弱。

我的评价：

- 对比表有用，但有明显事实风险。
- 它把 ChatPRD 的协作、历史版本、文档对比、团队共创等能力写得很满，但这些并没有在免费体验中验证。
- 对 pm-skills 的理解也可能不准。pm-skills 更像 agentic PM skills marketplace，不只是“题库 / 模板下载”。
- 这说明 ChatPRD 能生成看起来专业的竞品表，但容易把未验证推断写成事实。

### 问题 2-3：自我批判

ChatPRD 承认自己前一轮竞品分析存在问题：

太泛：

- 目标用户定位过宽。
- 输出质量没有评价标准。
- 工作流能力描述抽象。
- 差异化没有形成直接对比。

缺证据：

- 缺真实用户反馈。
- 缺产品实际数据。
- 缺输出样例对比。
- 缺市场分布和行业采用情况。

如果写成作品集，需要补充：

- 用户画像和痛点细分。
- 同一任务下各产品输出样例。
- 5 到 10 名真实用户反馈。
- 竞品功能和体验流程拆解。
- 场景化评测和选型建议。
- 量化结论和可落地建议。

我的评价：

- 这一轮很有价值，因为它能反思自己的不足。
- 这也暴露了 ChatPRD 的工作方式：首轮会给一个“貌似完整”的咨询式答案，追问后才会承认需要证据和样例。
- 对 AI PM Research Agent 来说，应该把“证据分层”和“未验证提示”作为默认输出，而不是等用户追问。

## 问题 3：GitHub 项目分析后的追问

项目链接：[addyosmani/agent-skills](https://github.com/addyosmani/agent-skills)

### 问题 3 首轮：产品分析报告

ChatPRD 对 agent-skills 的首轮分析包括：

- 一句话说明：为 AI 编程代理设计的工程技能框架。
- 目标用户：AI 编程代理、AI 助手平台开发者、技术团队、开源维护者。
- 核心场景：增强 AI 编程代理能力、自动代码生成、重构、PR 流程、工程治理。
- 痛点：AI 编程容易只求能跑，不懂生产级规范、测试、安全和可维护性。
- 产品亮点：覆盖软件开发生命周期、结构化技能、自动激活、强制测试和评审。
- 技术亮点：Markdown 技能规范、工作流路由、反 rationalization、失败暂停、多角色协作。
- 商业化方向：企业级工程守门人、AI IDE 集成、DevOps / AIOps 平台。
- 作品集方向：自动化 Pull Request 审核机器人、流程日志、质量报告、GitHub Actions 集成。

我的评价：

- 这次首轮质量高于前两题，因为 GitHub 项目本身信息密度高。
- 它能抓住 agent-skills 的核心产品价值：把工程方法论组件化给 AI agent 使用。
- 但它仍然把一些推断写成了事实，例如 OpenTelemetry、企业大规模落地、多 Agent 协作边界等。

### 问题 3-1：事实、推断、待验证

ChatPRD 将 agent-skills 分析拆为：

已验证事实：

- agent-skills 是生产级工程技能集合。
- 包含 23 项开发生命周期技能和 1 个元技能，共 24 个。
- 使用结构化 Markdown 定义技能。
- 支持按任务类型自动激活技能。
- 提供 `/spec`、`/plan`、`/build`、`/test`、`/review` 等 slash commands。
- 强调测试驱动、review-before-merge、可验证、可追溯和失败人工干预。
- 参考 Google 工程最佳实践。

合理推断：

- 目标用户也包括企业技术管理者和 DevOps 工具平台。
- 可以适配 CI/CD、代码审核机器人等企业定制场景。
- 有商业交付潜力。
- 可以启发 AI PM 做结构化工程治理产品。
- 作品集方向可以是自动化审核机器人或增强前后对比。

需要进一步验证：

- 多平台兼容性和集成难度。
- 与企业 DevOps / GitHub Actions 的真实整合效果。
- 生产中的可观测性实践反馈。
- 安全、质量等特殊场景是否足够灵活。
- 多 Agent 协作边界和冲突解决机制。

我的评价：

- 这是目前最接近“合格研究型回答”的一轮。
- 它开始明确区分事实和推断，也指出哪些内容要用实际体验验证。
- 不过仍然需要链接具体 README 段落、命令说明或文件路径，才能变成真正的 source-backed 分析。

### 问题 3-2：竞品对比

ChatPRD 对比了：

- agent-skills
- phuryn/pm-skills
- ChatPRD
- ChatGPT / Claude + 提示词

主要观点：

- agent-skills 面向“AI 变工程师”，强调生产级工程流程和规范。
- pm-skills 和 ChatPRD 聚焦产品和 PM 领域的结构化知识。
- ChatGPT / Claude + 提示词灵活，但标准化和专业流程弱。
- agent-skills 和 ChatPRD 更适合 B2B、团队订阅和平台集成。
- 通用提示词方案更适合 B2C 或超级用户市场。

我的评价：

- 这轮对“不同类型解决方案”的分层有价值。
- 但对 pm-skills 和 ChatPRD 的商业模式判断仍然偏推断。
- “prompt 方案是玩具型市场”这个说法偏武断，需要小心。强提示词方案可能是许多专业用户的真实工作流，不一定只是玩具。

### 问题 3-3：2 周作品集 MVP 计划

ChatPRD 给出的作品集方向：

项目名：AI 工程流程自动化守门人。

目标用户：

- 技术型产品经理。
- AI 产品与工程团队。
- 希望自动化工程流程、提升代码质量的小型开发组。

核心功能：

- 接入 agent-skills 到 Pull Request 流程。
- 自动触发 `/spec`、`/plan`、`/build`、`/test`、`/review` 等流程。
- 展示每个 PR 的技能流转、守门日志和质量报告。
- 发布 demo 仓库和作品集文档。

2 周计划：

- 第 1 周：需求梳理、接入 agent-skills、选择核心技能、搭建 demo 框架、采集 artifact、联测。
- 第 2 周：补充高阶用例、支持扩展技能入口、丰富反馈和可视报告、打磨 PR 流水线、录制 demo、整理 GitHub。

产出物：

- 需求方案文档。
- MVP demo 仓库。
- 自动生成的 SPEC.md、测试报告、review log。
- 使用说明和 demo 视频 / GIF。
- 展示用 slides 或 PDF。

评价指标：

- 产物数量和规范率。
- 全流程一次通过率。
- 平均审批耗时。
- 异常流程留痕数。
- 用户体验反馈。
- star / fork / issue 等作品集外部反馈。

我的评价：

- 这轮对作品集很有启发，但执行难度偏高。
- 它默认要做 GitHub Actions、前端 dashboard、PR 自动化和技能链路，这对 2 周个人 MVP 可能太重。
- 更合理的个人版 MVP 是：先选一个 demo repo，手动或半自动跑 3 个技能，生成对比报告和质量评分，而不是立刻做完整自动化守门人。

## 总体结论

### ChatPRD 多轮追问后明显变强

首轮输出通常是：

- 结构完整。
- 语言顺。
- 像咨询报告。
- 但偏模板化、证据弱、容易把推断写成事实。

多轮追问后，它能补出：

- 已验证事实 / 合理推断 / 待验证内容。
- 2 周 MVP 验证计划。
- 关键假设和最低成本测试。
- 自我批判。
- 竞品对比。
- 作品集落地方案。

### 但它仍有关键短板

1. 需要用户主动追问，才会暴露假设和证据缺口。
2. 不会默认引用来源和具体证据位置。
3. 容易把“合理推断”写成产品事实。
4. 计划常常偏完整工具，而不是最小验证实验。
5. 对竞品的事实细节可能不准，需要人工核验。

## 对 AI PM Research Agent 的产品启发

AI PM Research Agent 不应该只是“生成更漂亮的 PRD”。它应该默认做到 ChatPRD 需要被追问后才会做的事：

1. 每个判断标注类型：事实、推断、待验证。
2. 每个关键结论尽量附来源链接或证据位置。
3. 默认给出最危险假设和低成本验证方法。
4. 默认提供“首轮输出”和“追问后修正版”的差异。
5. 默认把项目分析转化为可执行作品集计划。
6. 默认提醒哪些内容太泛、哪些内容缺证据。

## 下一步建议

接下来可以做两件事：

1. 把 ChatPRD 的这次完整测试整理成竞品体验案例。
2. 设计 `AI PM Research Agent` 的第一个模板：`GitHub AI 项目产品分析模板`。

这个模板应至少包含：

- 项目一句话定义。
- 目标用户。
- 核心场景。
- 痛点。
- 产品亮点。
- 技术亮点。
- 商业化方向。
- 对 AI PM 的启发。
- 作品集转化方案。
- 事实 / 推断 / 待验证三栏。
- 最危险假设。
- 下一步手动体验建议。
